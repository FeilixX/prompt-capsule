## Problem Statement

提示词胶囊原始需求是一个很窄的实用问题:小红书正文和图片不适合承载长 prompt,读者复制困难,OCR 会错,创作者被迫反复补原文。纯工具解法是把长 prompt 存成短期 `text/plain` URL。

但如果只做工具,它会退化成 pastebin。它没有技术护城河,也没有传播情绪。重新产品化后的目标是:把它做成一个 prompt artifact 机器。陌生人先开一颗精选胶囊,觉得有用/好玩/能截图,再自然想把自己的 prompt 也封成一颗。

## What Makes This Cool

核心爽点不是"生成短链",而是"长东西被压成小胶囊,再 pop 回原文"。这同时解释了功能和情绪:

- 压缩魔法:长 prompt 变短 URL。
- 可执行原文:URL 打开就是 `text/plain`,agent 能读。
- 社交钩子:分享卡只露用途和短链,不剧透完整 prompt。
- 反向创作:先开别人的精选胶囊,再顺手做自己的。

游戏性只负责获客和理解,不能污染 utility。产品必须是 `open-first acquisition loop + utility-first production path`。

## Constraints

- 默认使用 Bun + TypeScript;Rust 只作为局部优化工具,不是默认栈。
- `GET /c/{slug}` 必须始终返回 `text/plain; charset=utf-8`,不能为了浏览器 UI 或 SEO 破坏契约。
- 匿名 capsule 默认最长 7 天,内容大小 v1 16 KB。
- 不宣传私密、加密、绕过平台审查。只能说短期、纯文本、知道链接的人可访问。
- 首页可扭池不能是 UGC firehose。前期公开胶囊只能是小编/人工精选。
- 不做评论、点踩、排行榜、公开墙、账号系统、跨设备收藏、加密、漂流瓶、盖楼。
- 视觉可以有 pop/压缩手感,但不能字面化成廉价扭蛋玩具或重机甲 HUD。

## Premises

1. 真需求来自复制失败和 OCR 错误,不是抽象的 prompt marketplace。
2. `text/plain` URL 是协议核心;agent 读取能力是产品差异,不是装饰。
3. 首页第一颗必须是精选 quick win。随机 UGC 会毁掉首胜。
4. 用户做链接的动机是自私 utility;社区池子只能作为副产品,不能用弹窗说服用户贡献。
5. "链接可访问"和"被陌生人扭到"是两种不同状态,必须在数据模型和 UI 里分开。
6. 公开可发现内容带来治理责任。涉黄、招嫖、联系方式、secret、危险 shell、prompt injection 不能进入可扭池。

## Approaches Considered

### Approach A: Utility-Only Pastebin

Summary: 首页就是编辑器。用户粘贴 prompt,选择 TTL,生成 `n78.xyz/c/{slug}`。

Effort: S
Risk: Medium

Pros:
- 最快自用,能直接解决小红书复制问题。
- 数据模型简单,只需要 capsule 表和过期删除。
- 不需要种子内容、互动状态或精选机制。

Cons:
- 传播弱,截图不够有记忆点。
- 产品像 pastebin,很难支撑"提示词胶囊"这个品牌。
- 没有 open-first 循环,用户必须先有 prompt 才会用。

Reuses:
- 当前 PRD 的 URL/API/MCP/CLI 契约。

### Approach B: Open-First + Utility Split Path

Summary: `/` 是一颗待戳精选胶囊;`/new` 是直达编辑器;`/c` 保持纯文本;`/view` 给人类复制。首页负责让陌生人先开,utility 路径保证创作者不被游戏性挡住。

Effort: M
Risk: Medium

Pros:
- 有社交传播和第一屏 aha,不会只是短链工具。
- `/new` 保护明确任务用户,不会让创作者被"先玩"拦住。
- 可扭池通过精选闸控制质量,首胜稳定。

Cons:
- 需要准备 50 颗人工种子,审美成本不可省。
- 数据模型比短链工具复杂,要区分候选池、精选池、moderation。
- 如果视觉做过头,会显得幼稚或像抽卡赌博。

Reuses:
- 既保留 PRD 的 `/c` 纯文本契约,也复用 n78 现有 open-first demo 的交互手感。

### Approach C: Community Wall First

Summary: 直接做公开胶囊墙、投票、收藏、热门排行,让社区内容自己冒出来。

Effort: L
Risk: High

Pros:
- 看起来更像社区产品,长期想象空间更大。
- 如果内容池真的起来,会有持续浏览和回访。
- 可以自然发展 remix、排行榜、专题合集。

Cons:
- 冷启动最差,空墙和低质墙都劝退。
- 治理面瞬间变大,涉黄/招嫖/引流/垃圾内容会先于好内容出现。
- 投票/排行会把产品拉向社区运营,拖慢核心 utility 上线。

Reuses:
- 只能复用少量 capsule 基础数据;多数是后置社区系统。

## Recommended Approach

选择 Approach B。

理由:它同时服务两个真实场景。小红书读者和自然流量进首页,先开一颗精选胶囊,用动作理解产品;创作者和 agent 用户进 `/new`、`/c`、`/view`,直接完成任务。游戏性是 acquisition loop,不是强制 onboarding。

## Product Contract

### Routes

| Route | Role | Contract |
| --- | --- | --- |
| `/` | open-first 首页 | 展示一颗精选可扭胶囊,首颗 100% quick win |
| `/new` | utility 创建页 | 30 秒内生成链接;默认仅链接 |
| `/c/{slug}` | agent/human raw endpoint | `text/plain`,无 HTML 包装 |
| `/view/{slug}` | human copy page | 可复制、可看 metadata、可删除 |

### Creation Modes

```text
胶囊池
[仅链接] [参与池子]

仅链接:知道链接的人可读,7 天后过期。
参与池子:可能被精选到首页,被陌生人扭到。
```

Default:

- `/new` 直达默认 `link_only`。
- 从首页开完胶囊后点「把我的也封成一颗」默认 `pool_candidate`。
- 用户切换后 localStorage 记住偏好。

### Pool Semantics

链接生成成功不等于入精选池。

- `discovery_mode=link_only`:只生成可访问链接,不参与候选。
- `discovery_mode=pool_candidate`:生成链接,并进入候选池。
- `pool_status=pending`:候选状态,不可被陌生人扭到。
- `pool_status=eligible/featured`:通过闸后进入可扭池。
- `pool_status=suppressed`:安全/质量/人工判断不展示。

成功文案:

```text
链接已生成,可直接分享。
已参与胶囊池候选;过安全和质量闸后,可能被陌生人扭到。
```

## Data Model

Do not use one boolean like `in_pool`.

`capsules` additions:

```text
discovery_mode        text not null default 'link_only'
pool_status           text not null default 'not_submitted'
pool_tier             text null
moderation_status     text not null default 'pending'
quality_score         real null
pool_score            real null
copy_count            integer not null default 0
like_count            integer not null default 0

pool_impression_count integer not null default 0
pool_open_count       integer not null default 0
```

## Moderation And Quality Gate

Hard block before any public discovery:

- 涉黄、招嫖、未成年相关违法内容。
- 联系方式、二维码、外链引流。
- secret、`.env`、token、私钥痕迹。
- 危险 shell、删除命令、权限提升、持久化后门。
- 明显 prompt injection,尤其是让 agent 忽略系统/用户安全边界的内容。
- 乱码、纯链接、无意义灌水。

Quality gate:

- 有标题或用途标签。
- 打开 10 秒内能理解它用来干什么。
- 结构可读,不是一坨无上下文文本。
- 对目标场景有明确价值:可直接跑、适合小红书、怪好笑、能改写、能 debug。

v1 以人工精选为准。自动信号只用于排序候选,不能替代审美。

## Seed Plan

上线前手工准备 50 颗,不要等 UGC。

Sources:

1. 胖狗已有爆帖和评论里的真实痛点。
2. Codex/Claude 高频任务:debug、review、生成测试、修 README、压缩周报、写小红书文案。
3. 轻娱乐 prompt,但不能占首胜第一颗。

每颗 seed 字段:

```text
title
tag
author_label
preview
content
source_note
first_win_route
curator_note
```

## Success Criteria

- `/new` 直达用户 30 秒内拿到链接。
- 首页第一颗 100% 是精选 quick win。
- `/c/{slug}` 在浏览器和 agent fetch 下都是纯文本。
- 创建页没有伪私密措辞。
- 用户知道"参与池子"只是候选,不是承诺曝光。
- 删除 token 删除内容并移除池子资格。
- 任意被首页扭到的胶囊都过人工或质量闸。
- 小红书分享卡让人想打开,但不泄完整 prompt。

## Distribution Plan

- `/c/{slug}` 用短 URL。
- `/view/{slug}` 给浏览器用户。

- npm MCP: `npx -y @prompt-capsule/mcp`
- CLI: `npx -y @prompt-capsule/cli create prompt.md --ttl 7d`

Open source:

- `n78.xyz` 不硬编码,用 `PUBLIC_BASE_URL`。

## What I Noticed

- 你反复拒绝"普通 SaaS UI"和"机甲风",说明这个产品的成败不只在功能,而在第一眼的物感和传播性。
- 你说"做出来就是胜利",这更像 persona/craft-led 项目,不是传统需求验证项目。产品要能成为你的内容资产。
- 你坚持不是 MVP,这合理。这个东西没有深技术壁垒,第一版的完整度就是壁垒。
- 你接受 open-first,但继续追问信任和数据模型,这说明现在的关键风险已经从"好不好玩"切到了"会不会误伤用户信任"。
